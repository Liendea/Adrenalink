ADRENALINK – Bokningssystem för extremsport entusiaster
Teknisk rapport – Arkitektur, Backend, API & ORM
=======================================================


PROJEKTÖVERSIKT
===============

Adrenalink är en fullstack-webbapplikation för bokning av extremsportlektioner.
Plattformen kopplar ihop elever med skolor och instruktörer i Europa, med fokus
på aktiviteter som surf, vindsurfing, kitesurfing, klättring, snowboard och
wakeboard. En inloggad användare kan söka efter och boka lektioner, betygsätta
skolor och spara favoriter.

Projektet är uppbyggt som ett kundsystem och kombinerar ett backend-slutprojekt
med ett fullstack-kundprojekt, där stor vikt har lagts på båda delarna. Frontend
är ett komplett, användarvänligt gränssnitt och backend är en välstrukturerad
Express-serverapplikation med RESTful API.

Teknisk stack:
  Backend:    Node.js + Express 5 + TypeScript + Prisma ORM + MySQL
  Frontend:   React 19 + React Router v7 + SCSS + Vite
  Databas:    MySQL med Prisma Migrate
  Auth:       JWT (JSON Web Tokens) via jsonwebtoken
  Geocoding:  OpenStreetMap Nominatim API


ARKITEKTUR
==========

Repostrukturen följer en layered architecture, där funktionalitet är separerad
efter ansvarsområde och lager. Projektet är uppdelat i två tydliga delar:
/Server (backend) och /client (frontend).

Backend-sidan är organiserad i distinkta lager:

  Routes       -> tar emot HTTP-förfrågningar och delegerar till rätt controller
  Controllers  -> hanterar request/response-logik och anropar Prisma
  Middleware   -> exekveras mellan route och controller (t.ex. autentisering)
  Config       -> databasanslutning och miljövariabler
  Utils        -> hjälpfunktioner (t.ex. geokodning)
  Types        -> TypeScript-typdeklarationer

Serverfilen (server.ts) ligger i src/-mappen, inte i roten. Det ger ett
tydligare källkodspaket och innebär att kompilering med tsc sker samlat
och till en gemensam /dist-mapp.

Katalogstruktur (backend):

  Server/
  ├── prisma/
  │   ├── schema.prisma         # Databasmodeller och relationer
  │   ├── seed.ts               # Testdata (skolor, lektioner, tidsluckor)
  │   └── migrations/           # Prisma Migrate-historik
  └── src/
      ├── server.ts             # Startpunkt: portlyssning och DB-uppstart
      ├── app.ts                # Express-app: middleware och routemontering
      ├── config/
      │   └── db.ts             # Prisma-klientsingleton och connectDB()
      ├── controllers/
      │   ├── authController.ts
      │   ├── schoolController.ts
      │   ├── slotController.ts
      │   ├── ratingController.ts
      │   ├── exploreController.ts
      │   └── favoriteController.ts
      ├── middleware/
      │   └── authMiddleware.ts
      ├── routes/
      │   ├── authRoutes.ts
      │   ├── schoolRoutes.ts
      │   ├── slotRoutes.ts
      │   ├── ratingRoutes.ts
      │   ├── exploreRoutes.ts
      │   └── favoriteRoutes.ts
      ├── types/
      │   └── express.d.ts      # Module augmentation för req.user
      └── utils/
          └── geocoding.ts      # OpenStreetMap Nominatim-integration

Express-appen monterar routes under ett gemensamt /api/-prefix:

  app.use('/api/auth',      authRoutes)
  app.use('/api/explore',   exploreRoutes)
  app.use('/api/slots',     slotRoutes)
  app.use('/api/schools',   schoolRoutes)
  app.use('/api/ratings',   ratingRoutes)
  app.use('/api/favorites', favoriteRoutes)

Globala middleware:
  cors()          -> tillåter anrop från http://localhost:5173 med credentials
  express.json()  -> parsar JSON-body i inkommande requests


ORM – PRISMA
============

Som ORM används Prisma med MySQL-databas. Prisma hanterar databasschemat
via schema.prisma, och ändringar versionshanteras med Prisma Migrate. Det ger
typade databasklienter, automatisk validering och parametriserade SQL-frågor
som eliminerar risken för SQL-injektion.

Prisma-klienten initieras som en singleton i config/db.ts och återanvänds
i alla controllers via import.

ref: https://www.prisma.io/docs


DATABAS & SCHEMA
================

Provider: mysql
Databas:  adrenalink


Modell: User
  Beskrivning: Representerar en registrerad användare. Rollen styr behörighet
  och kan vara elev, instruktör eller administratör. Profilbilden lagras som
  Base64-kodad sträng direkt i databasen (LongText).

  Fält:
    id            Int       PK, auto-increment
    email         String    Unik
    password      String    Hashad med bcrypt
    firstName     String
    lastName      String
    passportNo    String    Unik
    address       String
    zipCode       String
    city          String
    country       String
    phoneCode     String    T.ex. "+46"
    phoneNumber   String
    role          Enum      STUDENT | INSTRUCTOR | ADMIN  (default: STUDENT)
    profileImage  String?   Base64-kodad, LongText
    createdAt     DateTime  auto-satt
    updatedAt     DateTime  auto-satt

  Relationer: ratings[], favorites[]
  Databastabell: users


Modell: School
  Beskrivning: Representerar en extremsportskola med geografisk plats.
  Koordinaterna (lat/lng) beräknas via OpenStreetMap Nominatim vid seeding
  och möjliggör kartvisning i frontend.

  Fält:
    id        Int       PK, auto-increment
    name      String
    country   String
    city      String
    address   String
    lat       Float?    Latitud (geokodad)
    lng       Float?    Longitud (geokodad)
    createdAt DateTime  auto-satt

  Relationer: lessons[], ratings[], favorites[]
  Databastabell: schools


Modell: Lesson
  Beskrivning: Representerar en specifik lektionstyp på en skola. En lektion
  kan ha flera tidsluckor (AvailableTime) och kan bokmärkas av användare.
  Sport- och lektionstyp definierar vad som erbjuds och på vilken nivå.

  Fält:
    id                Int      PK, auto-increment
    schoolId          Int      FK -> School (cascade delete)
    lessonType        String   'group' | 'private'
    sportType         String   'surf' | 'windsurf' | 'kitesurf' |
                               'snowboard' | 'wakeboard' | 'climbing'
    description       Text
    level             String   'beginner' | 'intermediate' | 'advanced'
    durationHours     Float
    priceEuro         Float
    location          String
    equipmentIncluded Boolean  default: true
    lat               Float?
    lng               Float?

  Relationer: school (FK), availableTimes[], favorites[]
  Databastabell: lessons


Modell: AvailableTime
  Beskrivning: En specifik tidslucka för en lektion. isBooked-flaggan sätts
  till true när en bokning görs, vilket döljer platsen från sökresultat.
  Tidsluckorna är seedade till november–december 2026.

  Fält:
    id        Int       PK, auto-increment
    lessonId  Int       FK -> Lesson (cascade delete)
    startTime DateTime
    isBooked  Boolean   default: false

  Relationer: lesson (FK)
  Databastabell: available_times


Modell: Rating
  Beskrivning: Betygsättning av skola av en inloggad användare. En användare
  kan sätta ett betyg per skola, och uppdatering hanteras med Prismas
  upsert-mönster. Genomsnittligt betyg beräknas vid hämtning.

  Fält:
    id        Int       PK, auto-increment
    schoolId  Int       FK -> School (cascade delete)
    userId    Int       FK -> User (cascade delete)
    score     Int       1–5
    createdAt DateTime
    updatedAt DateTime

  Unik constraint: [schoolId, userId]  (en röst per användare och skola)
  Relationer: school (FK), user (FK)
  Databastabell: ratings


Modell: Favorite
  Beskrivning: En användare kan spara antingen en lektion eller en skola som
  favorit (aldrig båda i samma rad – lessonId och schoolId är valfria). Unika
  constraints förhindrar dubbletter. Favoriter hämtas med inbäddat lesson.school
  för att ge fullständig information i en enda Prisma-query.

  Fält:
    id        Int       PK, auto-increment
    userId    Int       FK -> User (cascade delete)
    lessonId  Int?      FK -> Lesson (cascade delete, valfritt)
    schoolId  Int?      FK -> School (cascade delete, valfritt)
    createdAt DateTime  auto-satt

  Unika constraints: [userId, lessonId], [userId, schoolId]
  Relationer: user (FK), lesson? (FK), school? (FK)
  Databastabell: favorites


Relationsöversikt:

  User       1──* Rating     (en användare kan betygsätta många skolor)
  User       1──* Favorite   (en användare kan ha många favoriter)
  School     1──* Lesson     (en skola erbjuder många lektioner)
  School     1──* Rating     (en skola kan ha många betyg)
  School     1──* Favorite   (en skola kan sparas av många användare)
  Lesson     1──* AvailableTime
  Lesson     1──* Favorite


RESTFUL API – ENDPOINTS
=======================

Alla endpoints är samlade under /api/-prefixet.


Auth – /api/auth
----------------

POST   /api/auth/register
  Registrerar ny användare. Validerar att email och passportNo är unika.
  Hashar lösenord med bcrypt (10 rounds).
  Body:  { email, password, firstName, lastName, passportNo,
           address, zipCode, city, country, phoneCode, phoneNumber }
  Svar:  { userId }  (status 201)

POST   /api/auth/login
  Loggar in befintlig användare. Jämför lösenord med bcrypt.
  Utfärdar JWT-token med payload { userId, role }, giltig i 1 dag.
  Body:  { email, password }
  Svar:  { token, user }

POST   /api/auth/profile-image
  Uppdaterar profilbild för angiven användare.
  Body:  { userId, imageBase64 }
  Svar:  uppdaterat user-objekt


Schools – /api/schools
----------------------

GET    /api/schools/
  Hämtar alla skolor. Inkluderar lektioner och betygsdata (ratings[].score).
  Sorteras i bokstavsordning på namn.
  Query: ?country=<string>  (valfritt filter)
  Svar:  { schools }


Slots (AvailableTime) – /api/slots
-----------------------------------

GET    /api/slots/lesson/:lessonId
  Hämtar en lektion med tillhörande lediga tidsluckor.
  Inkluderar: school, availableTimes (isBooked=false, sorterat på startTime).
  Params: lessonId (number)
  Svar:   lektionsobjekt med availableTimes

GET    /api/slots/?lessonId=X
  Hämtar alla lediga tidsluckor för en specifik lektion.
  Query:  lessonId (number)
  Svar:   array av available_times (isBooked=false)

GET    /api/slots/:id
  Hämtar en enskild tidslucka med inkluderad lesson.school.
  Params: id (number)
  Svar:   tidsluckeobjekt eller 404


Ratings – /api/ratings
-----------------------

GET    /api/ratings/schools/:schoolId
  Hämtar genomsnittligt betyg och totalt antal betyg för en skola.
  Params: schoolId (number)
  Svar:   { average, count }

POST   /api/ratings/schools/:schoolId
  Skapar eller uppdaterar betyg (Prisma upsert).
  En användare, ett betyg per skola.
  Params: schoolId (number)
  Body:   { userId, score }  (score: 1–5)
  Svar:   { rating, average, count }


Explore – /api/explore
-----------------------

GET    /api/explore/
  Sökendpoint med filtreringsstöd. Returnerar tillgängliga tidsluckor med
  inbäddad lektion och skola. Filtrera på land, plats och datumintervall.
  Query:
    country    (string, valfritt; "Nearby" är ett specialfall)
    location   (string, valfritt; delsträngssökning)
    startDate  (ISO-datum, valfritt)
    endDate    (ISO-datum, valfritt)
  Svar:  { message, criteria, availableSlots }


Favorites – /api/favorites  [kräver autentisering]
----------------------------------------------------

GET    /api/favorites/
  Hämtar alla favoriter för inloggad användare.
  userId hämtas automatiskt från JWT-token via authMiddleware.
  Inkluderar: lesson.school, school. Sorteras på createdAt (nyaste först).
  Header: Authorization: Bearer <token>
  Svar:   { favorites }

POST   /api/favorites/
  Skapar ny favorit (lektion eller skola). userId hämtas från token.
  Header: Authorization: Bearer <token>
  Body:   { lessonId? } eller { schoolId? }
  Svar:   { favorite }  (status 201)

DELETE /api/favorites/:id
  Tar bort en specifik favorit.
  Validerar att favoriten tillhör den inloggade användaren.
  Header: Authorization: Bearer <token>
  Params: id (favorite ID)
  Svar:   { message }


MIDDLEWARE
==========

authMiddleware.ts – authenticateToken()
  Extraherar Bearer-token från Authorization-headern.
  Verifierar token med JWT_SECRET via jsonwebtoken.
  Avkodar payload till { userId, role } och tillgängliggör det som
  req.user = { id, role } på request-objektet.
  Returnerar 401 om token saknas, 403 om token är ogiltig eller utgången.

  Används på:
    GET    /api/favorites/
    POST   /api/favorites/
    DELETE /api/favorites/:id


TYPSYSTEM & TYPESCRIPT
======================

TypeScript används i hela backend med strict mode aktiverat. Prisma genererar
automatiskt typer från schema.prisma, vilket ger full typning av
databasklienten och reducerar risken för runtime-fel.

Express module augmentation (src/types/express.d.ts):
  Utökar Express Request-interface med:
    user: { id: number; role: string }
  Det möjliggör typat tillträde till req.user i alla controllers utan
  explicit typcastning, tack vare TypeScripts module augmentation och
  typeRoots-konfigurationen i tsconfig.json.

TypeScript-konfiguration:
  Module:        nodenext  (ESM-kompatibel)
  Target:        esnext
  Strict mode:   aktiverat
  Source maps:   aktiverade
  typeRoots:     ./src/types, ./node_modules/@types

Kompileringssteg:
  Dev:    tsx watch src/server.ts   (hot reload utan separat kompileringssteg)
  Build:  tsc                       (kompilerar TypeScript till /dist)
  Start:  node dist/server.js


AUTENTISERING & SÄKERHET
=========================

Lösenordshantering:
  Lösenord hashas med bcrypt (10 salt rounds) vid registrering och
  jämförs säkert med bcrypt.compare() vid inloggning. Det hashade
  lösenordet returneras aldrig i API-svaret.

JWT-autentisering:
  Token utfärdas vid inloggning med payload { userId, role }.
  Utgångstid: 1 dag.
  Skyddade routes verifieras av authenticateToken-middleware.

SQL-injektion:
  Prisma ORM genererar parametriserade SQL-frågor och eliminerar
  risken för SQL-injektion i samtliga databasoperationer.

CORS:
  Begränsat till en enda tillåten origin: http://localhost:5173.
  credentials: true tillåter att Authorization-headern skickas från frontend.

  mellan varje API-anrop för att respektera Nominatims rate limit.


FRONTEND
========

Även om rapportens fokus är backend, är frontenddelen av stor vikt då
projektet är ett fullstack-kundprojekt. Stor arbetsinsats har lagts på att
skapa ett komplett och användarvänligt gränssnitt parallellt med
backend-utvecklingen.

Frontend är byggt med React 19, React Router v7 och SCSS och kommunicerar
med backend-API:et via fetch-anrop kapslade i egna React hooks.

Sidor och routes (client/src/App.tsx):
  /                   -> LandingPage          (publik)
  /register           -> Register             (publik)
  /login              -> Login                (publik)
  /explore            -> ExplorePage          (publik)
  /booking/:lessonId  -> BookingPage          (publik)
  /schools/:schoolId  -> SchoolDetailPage     (publik)
  /profile            -> UserProfile          (skyddad)
  /profile/edit       -> EditProfile          (skyddad)
  /profile/favorites  -> MyFavorites          (skyddad)
  /profile/bookings   -> MyBookings           (skyddad)

Skyddade routes hanteras av en ProtectedRoute-komponent som kontrollerar
autentiseringsstatus via AuthContext.

Frontend-arkitektur:
  Hooks       -> kapslar API-anrop och state
                 (useActivitySearch, useFavorites, useLogin, useRegister, m.fl.)
  Context     -> global state för auth (AuthContext) och favoriter (FavoritesContext)
  Components  -> återanvändbara UI-komponenter (Card, Map, Rating, SearchBar,
                 Navigation, Footer)
  Pages       -> sidkomponenter som komponerar hooks och context

Kartintegration:
  MapLibre GL och react-map-gl används för att rendera en interaktiv karta
  på ExplorePage. Skolor visas som punkter baserat på de lat/lng-koordinater
  som geokodades vid seeding.
